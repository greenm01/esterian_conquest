# Project Manual Style Guide (Old-School Edition)

## Purpose
Keep the manuals clear, direct, and in the spirit of classic game documentation — straightforward, no-nonsense, like old printed booklets and MUD/sysop guides.

## Voice and Tone
- **Player manuals**: Address the reader as "you". Use straightforward masculine defaults: he, him, his when referring to the player or a generic person. Speak plainly, like one gamer explaining the game to another.
- **Admin/sysop manuals**: Use clear, imperative language. Be authoritative and precise — this is written for the men who run the server.
- Overall: Short sentences. Active voice. Cut unnecessary words. Write as if it's 1992. Be complete but concise.

## Core Principles
- Clarity first. If something is confusing, rewrite it.
- Prefer simple, strong, traditional terms: master, sysop, administrator, etc.
- Default to masculine forms (he/him/his) for the player and administrator in general descriptions.
- Number every procedural step, especially in admin sections.
- Warnings and notes should be short and blunt. Start with WARNING:, NOTE:, or IMPORTANT:.

## Formatting Conventions
- **Commands**: Present as `COMMAND` or /command. Stay consistent.
- **Key terms**: **Bold** on first use when helpful (e.g., **Master Account**, **SysOp**).
- **Examples**: Keep them practical and real.

## Terminology
Use consistent traditional terms. Default to masculine pronouns for general cases.

Examples to follow:
- The player controls his character...
- Only the master sysop may do this. He edits the configuration file...

Add your game-specific terms here as you decide them (e.g., hit points, realm, etc.).

## Reference Hierarchy
1. This project style guide (old-school rules first)
2. Microsoft Writing Style Guide — for clear procedures and technical formatting (practical parts only)
3. Chicago Manual of Style — fallback for grammar, punctuation, and usage only

When in doubt: Does this read like a solid classic game manual or old sysop guide? If yes, keep it.
